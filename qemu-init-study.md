## 本章主线图

```text
启动命令
  -> main
    -> qemu_init
      -> qemu_create_machine
        -> 选择并创建 virt Machine
          -> class_init: 定义机器能力和默认行为
          -> instance_init: 创建具体实例和默认对象
        -> machine_done / 固件加载阶段
          -> 加载 OpenSBI / 客户机程序到内存
        -> vCPU reset
          -> PC 从 0x1000 起步
          -> 跳转到 OpenSBI 正式入口
      -> 进入 qemu_main_loop
        -> 主线程调度
        -> IO 线程处理 IO 事件
```

🧠 我的理解：你后面学别的架构、别的板级模型、别的设备时，也可以沿这条线去拆，不需要每次都从一大坨源码里瞎摸。

## 1. 为什么从这条启动命令入手

📘 讲义：文档用的最小示例是：

```bash
qemu-system-riscv64 -M virt -s -S -nographic
```

🎥 视频/作者：讲者特别强调，这条命令同时满足两个条件：
- `可观测`
- `可控`

它适合教学和调试，因为：
- `-M virt`：选一个最常见的 RISC-V 虚拟机器模型
- `-s`：启动 gdbstub，在 `1234` 端口等待 GDB
- `-S`：启动后先暂停 guest CPU
- `-nographic`：不弹图形窗口，直接把输出放到当前终端

🧠 我的理解：这几个参数配在一起，实际上是在给你创造一个“不会跑太快、而且你能看得见”的启动现场。

🧠 我的理解：为什么这很重要？
- 不暂停的话，QEMU 很快就跑过去了，启动前几步你根本抓不到
- 没终端输出的话，很多现象你也看不见
- 没 GDB 的话，你只能猜它做了什么，不能一步一步验证

✅ 你现在就做：
- 把这条命令记成“分析启动流程的教学命令”，不是普通运行命令。

## 3. virt Machine 是怎么被创建出来的

📘 讲义：文档里给出的思路是，从 `virt Machine` 的 `TypeInfo` 和相关初始化函数切进去，通过 GDB 打断点、看调用栈，去反推 QEMU 的启动主线。

🎥 视频/作者：讲者建议给两个函数下断点：
- `virt_machine_class_init`
- `virt_machine_instance_init`

然后通过 `bt` 看调用栈，抓到类似这条主线：

```text
main
  -> qemu_init
    -> qemu_create_machine
      -> 选择和创建 Machine
        -> class_init
        -> instance_init
```

🧠 我的理解：这一步最重要的不是“背住函数名”，而是理解 `class_init` 和 `instance_init` 的分工。

### `class_init` 干什么

🎥 视频/作者：讲者的总结很直接：
- `class_init` 更像是在定义机器的能力和默认行为

🧠 我的理解：你可以把它理解为“定义规则”。

这里更像是在说：
- 这台机器默认支持什么
- 默认 CPU 类型是什么
- 有哪些初始化回调
- 这个类型在 QEMU 的对象体系里具备什么能力

### `instance_init` 干什么

🎥 视频/作者：讲者说，`instance_init` 更像是在创建一个“具体实例”，做实例级初始化。

🧠 我的理解：你可以把它理解为“按规则造出一台具体机器”。

比如：
- 创建具体对象
- 初始化默认字段
- 准备 pflash 相关对象
- 让这台机器真正具备实例状态

### 这两层怎么记

🧠 我的理解：
- `class_init`：定义类型层面的规则
- `instance_init`：创建对象层面的实例

你先别纠结每一行源码细节，先把这个分层感建立起来。

✅ 你现在就做：
- 以后看到 QOM 相关初始化时，第一反应先问：
  - 这是在定义类型能力？
  - 还是在初始化具体实例？

## 4. 机器初始化的主线是什么

🎥 视频/作者：讲者强调了一点，`virt` 的初始化函数虽然长，但主线其实不乱。

视频里总结出来的节奏大致是：
- 先把硬件框架搭起来
- 做中断控制器等核心部件初始化
- 做地址空间/内存相关初始化
- 准备好设备树等信息
- 最后进入固件和客户机程序加载阶段

🧠 我的理解：这跟“先上电，再配硬件，再交给固件”的直觉是吻合的。

🧠 我的理解：你看这种初始化函数时，不要从第一行硬啃到最后一行。更好的办法是按“阶段”看：

1. 先搭框架
2. 再布置核心硬件资源
3. 再准备内存和设备信息
4. 最后再把固件/程序装进去

这样你看长函数就不容易迷路。

## 5. 固件和客户机程序是什么时候加载的

📘 讲义：文档明确说，观察完 `virt Machine` 创建之后，再去看 QEMU 如何把 OpenSBI 和客户机程序加载到内存里。

🎥 视频/作者：讲者特别强调：

- 固件不是一开始就急着加载
- 而是等整个 Machine 的内存空间和设备布局准备好以后
- 才进入固件和客户机程序加载阶段

🧠 我的理解：这很合理。

因为在固件加载之前，QEMU 至少得先知道：
- 这是什么 Machine
- 内存怎么布局
- 设备地址空间怎么安排
- 固件和 guest 程序该放哪

否则“往哪里加载”这个问题都还没被定义好。

🎥 视频/作者：视频里把关键路径概括成：

```text
machine 完整准备
  -> machine_done
    -> RISC-V 固件加载路径
      -> 根据 -bios/default 决定固件来源
      -> 加载 OpenSBI
```

🧠 我的理解：你可以把 `machine_done` 理解成一个阶段切换点：

- 前面是在“搭机器”
- 后面是在“往这台机器里装启动内容”

这是非常值得记住的分界线。

## 6. 为什么不写 `-bios` 也能跑

📘 讲义：文档明确写到，对于 `virt`（以及 `sifive_u`）这类机器，不指定 `-bios` 时，默认等价于 `-bios default`，会自动加载随 QEMU 发布的默认 OpenSBI。

🎥 视频/作者：讲者也在视频里提醒，高版本的 QEMU 才有这个默认行为，老版本不一定。

🧠 我的理解：这会直接影响你的调试判断。

如果你没写 `-bios` 但系统还是跑起来了，不代表“没有固件”，而是：

- QEMU 帮你选了默认固件
- 这个默认固件通常就是 OpenSBI

所以你后面追固件加载路径时，不能被“命令行里没写 `-bios`”误导。

✅ 你现在就做：
- 以后分析 `virt` 的启动问题时，先确认你当前 QEMU 版本对 `-bios default` 的行为。

## 7. vCPU 为什么先从 `0x1000` 跑

📘 讲义：文档给出了很关键的一段观察方法：
- 先用 GDB 连接上 QEMU 的 gdbstub
- 看 PC 在哪里
- 单步跟下去

文档里的现象是：

```bash
riscv64-elf-gdb -ex "target remote localhost:1234"
0x0000000000001000 in ?? ()
(gdb) display /i $pc
=> 0x1000: auipc t0,0x
```

接着继续单步，会跳到 OpenSBI 的正式入口地址。

🎥 视频/作者：讲者在视频里反复强调：
- 第一条指令先从 `0x1000` 开始
- 再跳到 OpenSBI 入口
- 这不是偶然

🧠 我的理解：这里最重要的不是“记住一个地址”，而是理解两个层次：

1. `0x1000` 是 reset 之后的起步位置
2. 它不是最终固件主体，只是一个很短的跳转入口

然后它再把执行流带到 OpenSBI 的正式入口。

### 怎么理解这段跳转

🎥 视频/作者：讲者用了一个很好的比喻：
- 你可以把 `0x1000` 附近那几条代码理解成“很早期的启动跳板”

🧠 我的理解：这个比喻我建议你记住。

更直白一点说：
- `0x1000` 像一小段前置引导/跳板代码
- 它的任务不是完成完整固件初始化
- 而是把控制权交给真正的 OpenSBI 入口

🧠 我的理解：所以你不要把“第一条指令在 `0x1000`”误解成“OpenSBI 从 `0x1000` 展开全部逻辑”。真正重要的是：

> reset 向量 -> 跳板代码 -> OpenSBI 正式入口

✅ 你现在就做：
- 看到 `0x1000` 时，不要只记地址，要记“它是 reset 后的起步跳板”。

## 8. 怎么把“现象”和“源码”对起来

🎥 视频/作者：讲者特别强调，`-s -S` 的组合非常适合学习启动流程，因为你可以从 guest 的第一条指令开始观察。

🧠 我的理解：这其实是这章最值钱的方法之一：

### 从“现象”出发
- GDB 连上去
- 看到 PC 在 `0x1000`
- 单步看到它跳到 OpenSBI

### 再回到“源码”
- 去看 CPU reset 路径
- 去看 reset vector 是在哪儿注册/赋值的
- 去看谁把它写成了那个值

🧠 我的理解：这就是很标准的“从运行现象反推源码责任点”的方法。你以后查：
- 启动异常
- guest 卡死
- 固件没进去
- 设备初始化时序问题

都可以用这招。

## 9. 主循环和 IO 线程到底在干什么

📘 讲义：文档把后半段收在“主循环与线程模型”里，重点提了：
- 主循环
- IO 线程

🎥 视频/作者：讲者把主循环概括成一个“事件泵”。

视频里的意思可以压缩成：
- QEMU 进入主循环后，不断处理事件
- 包括定时器、设备、monitor 等相关事件
- 直到满足退出条件才结束

🧠 我的理解：这个类比很实用。你可以把 `qemu_main_loop` 想成：

> 一个永远在“等事件 -> 处理事件 -> 再等事件”的中心调度器

### IO 线程是什么

🎥 视频/作者：讲者说，IO 线程的核心作用，是把 IO 相关事件从主线程里拆出去，放到独立上下文里处理。

视频里强调了两个好处：
- 减轻主线程压力
- 对 IO 密集路径提供更好的并发处理能力

🧠 我的理解：这一段你不用一开始就深挖到线程细节，先记住职责分工就够了：

- 主线程：更偏全局调度
- IO 线程：更偏 IO 事件处理

🧠 我的理解：如果你后面做的是块设备、网络设备、virtio、异步 IO，理解这层线程分工会很有帮助。

### 一句话总结

🎥 视频/作者：
- 主线程负责全局调度
- IO 线程负责 IO 事件

🧠 我的理解：这个总结很够用了，先记住，不要一上来陷进线程实现细节里。

## 11. 推荐的最小动手步骤

✅ 你现在就做：

1. 用这条命令启动 QEMU：

```bash
qemu-system-riscv64 -M virt -s -S -nographic
```

2. 开一个 GDB 连接上去：

```bash
riscv64-elf-gdb -ex "target remote localhost:1234"
```

3. 观察 PC 和当前指令：

```gdb
display /i $pc
si
si
si
```

4. 记录两个现象：
- 第一个停在哪个地址
- 跳到了哪个正式入口

5. 再去看讲义里提到的函数和主线，试着回答：
- 机器是什么时候创建出来的

- 为什么 CPU 不是直接从 OpenSBI 主体第一行开始跑
